# 《记忆的缝隙》关卡设计文档

> 版本：v0.1
> 地图工具：Tiled
> 网格：64×64 逻辑像素
> 相机设计分辨率：1280×720
> 对应 GDD：v0.1

## 1. 关卡设计原则

1. 每个章节只引入一种主要退化，不同时让玩家失去多个新能力。
2. 主要路径在不打开地图的情况下也能通过构图、物件、声音或纹理辨认。
3. 每个关键谜题至少包含两种不同类别的线索。
4. 走错或摆错不清空已经确认的正确步骤。
5. 检查点恢复后，角色必须位于安全导航区且面向当前目标。
6. 红伞只标记正确方向、关系记忆或安全空间，不用于误导。
7. 所有计时都用于提示升级，不用于倒计时失败。

## 2. 通用 Tiled 规范

地图图层、对象属性和命名以 TECHNICAL_DESIGN.md 为准。

### 2.1 坐标

本文坐标均为 Tiled 网格坐标，格式为 (x, y)，左上角为 (0, 0)。实体实际锚点为格子底部中心。

### 2.2 稳定 ID

| 前缀 | 用途 | 示例 |
| --- | --- | --- |
| map. | 地图 | map.home |
| spawn. | 出生点 | spawn.home.bed |
| entity. | 可见实体 | entity.home.key_bowl |
| interact. | 交互配置 | interact.home.key_bowl |
| trigger. | 一次性或区域触发 | trigger.home.thunder |
| puzzle. | 谜题 | puzzle.rain.station_sequence |
| checkpoint. | 存档点 | checkpoint.home.journal |
| dialogue. | 对白序列 | dialogue.home.wake |
| journal. | 日记页 | journal.home.key |
| memory. | 记忆碎片 | memory.rain.umbrella |
| audio. | 声音 | audio.rain.clock_bell |
| visual. | Tiled 可视化对象 | visual.home.bed |

> `visual.` 前缀对象仅存在于 `visual_furniture`、`visual_decor`、`visual_props` 层中。它们可作为运行时视觉数据来源，但不进入存档、不决定谜题真值，也不得替代 `interactables`、`collision`、`navigation` 等逻辑层的稳定 ID。

### 2.3 Tiled 运行时适配

`src/game/content/tiledMapLoader.ts` 是 Tiled JSON 与游戏运行时之间的唯一适配层。GameScene 在切章时调用 `parseTiledMap()` 将 Tiled 对象层转换为纯 TypeScript 数据结构：

- `interactables` → `WorldEntity[]`（坐标、kind、label）
- `collision` → `NamedCollisionRect[]`（纯 `AxisAlignedRect`）
- `visual_furniture` / `visual_decor` / `visual_props` → `VisualPlacement[]`（assetKey、frame、x/y、size、sortY、entityId、collisionId）
- `navigation` → `SpawnPoint[]` 和 `MovementBounds`

`visual_props` 通过 `entityId` 绑定 `interactables` 层实体，例如 `visual.home.key_bowl` 绑定 `entity.home.key_bowl`。`visual_furniture` 通过 `collisionId` 绑定 `collision` 层脚印矩形，例如 `visual.home.sofa` 绑定 `collision.home.sofa`。Tiled tile object 使用左下角坐标；适配层转换为 Phaser Image 的中心坐标，关卡作者不需要在 Tiled 中手动抵消偏移。

如果 Tiled JSON 缺少 visual_\* 层或解析失败，GameScene 自动回退到 `homeLayout.ts` 代码常量。GameStore 的碰撞和行走边界通过 `CollisionDataProvider` 注入；生产入口使用 `TiledCollisionProvider`，`CodeCollisionProvider` 仅作为测试或明确 fallback 使用。

### 2.4 Placeholder visual_props 规范

非 home 地图中暂无正式 tile 资产的 visual_props 对象使用 placeholder 模式。placeholder 对象没有 gid，不渲染 tile 图像，但仍携带 entityId/sortY/size 元数据供运行时深度排序和实体绑定使用。

**必须属性：** `placeholder=true`、`status=visual-placeholder`、`replacement=<非空资产说明>`、`entityId=<真实 interactables ID>`。

**禁止：** 有 gid 的正式 tile object 不得标记 `placeholder=true`。

`scripts/validate_tiled_maps.mjs` Check 7 强制校验上述规则。替换 placeholder 为正式资产时，在 Tiled 中为对象分配 gid 并移除 placeholder/status/replacement 属性。

由 Phaser 在运行时选择动画纹理的角色锚点允许不分配 gid，但必须使用 `status=actor-bound` 并绑定真实 `entityId`；这类对象不是待替换占位图。例如尾声的 `visual.ending.xiulan` 由 `character.xiulan_old.reach_hand.right` 提供正式动画。

### 2.5 通用交互距离

- 正面交互：角色锚点距离物件锚点不超过 96 像素；
- “静静留意”：只加强角色周围 192 像素内当前可见的已有线索；
- 门与出口：角色碰撞体进入 48 像素深的触发区域；
- 放置槽：打开物品选择面板后冻结移动。

### 2.6 通用进展回应

雨站序列与两个伞路标的每一步、Shared Life 每件物品归位、第四章每段正确前缀与每个路口完成、新增记忆片段、尾声牵手完成，均属于有效进展。每次有效进展在发生位置播放 0.6–1.2 秒局部场景回应和一个短三音锚点；只承认状态已经向前变化，不显示评分，不把回应解释为“恢复记忆”，也不改变提示计时、答案或检查点。静音路径必须仅凭场景回应成立。

## 4. 第一章：清晨的家

### 4.1 地图参数

- 地图 ID：map.home
- 尺寸：32×18 格
- 退化阶段：D0
- 出生点：spawn.home.bed = (5, 5)，朝下
- 相机：按房间 camera_zones 柔性切换

### 4.2 平面布局

~~~text
┌──────────┬────────────┬────────┐
│  卧室    │    客厅    │  厨房  │
│ P  照片  │ 日记 眼镜盒│        │
│          │            │        │
├──────┬───┴────────────┴────┬───┤
│储物间│       走廊           │玄关│
│封闭  │                      │碗 门
└──────┴──────────────────────┴───┘
~~~

储物间仅作为背景，不可进入，防止首章探索范围膨胀。

住宅建筑壳层 v11 沿用 v10 的全部正交墙体、房间开口和碰撞注册，只升级木地板尺度、墙面颗粒、材质磨损与接触阴影，使卧室、客厅、厨房、封闭储物间、走廊和右下玄关保持同一手绘空间语言。背景层包含地板、墙体、房间开口、右上四格窗、右下玄关地面，以及嵌入玄关右侧墙的门洞、门框和门槛；窗外阳光束、厨房地面亮斑、窗棂影子和跨房间的低透明度晨光反射由独立 `environment.home.sunlight_overlay` 绘制。入户门当前只作为出口 hotspot，不渲染独立门板 sprite。

住宅家具底座与墙体写入 `collision` 对象层，角色使用脚部小矩形碰撞，不允许进入家具底座；碰撞框只覆盖按 `homeVisualSizes` 校准后的落地脚印，不覆盖家具整张图片。v10 上方两道隔墙碰撞分别在 y=258、y=264 前结束；中墙从 y=342 开始，中墙碰撞区为 y=342–410，中央通道为 x=507–697。左下竖墙拆为上下两段，y=460–596 为储物间与中央走廊之间的通行开口。中墙、前墙和左下竖墙段的可见像素由透明 architecture overlay 按 y=410 与 705 参与排序；上方两道隔墙为竖向墙体，角色不会走到其后方，不再使用遮挡 overlay。住宅角色脚部可见像素与碰撞体底边对齐，穿过开口时身体由墙体遮挡，不再压在墙面上。

右下出口只作为室内侧交互点，不要求角色走进门洞：右侧外墙以 y=20–342、y=410–504 和 y=504–642 三段碰撞封闭，玩家在门内侧靠近 `(1225, 560)` 的 hotspot 即可触发出门交互。这样避免右墙门洞遮挡层与角色排序复杂化，同时仍保留嵌墙门框的视觉结构。

非交互装饰层在床尾放置旧条纹地毯与布拖鞋，在厨房台面放置搪瓷暖水瓶、调味罐和旧砧板，在茶几下放置低饱和旧地毯，在右下玄关放置门垫，并在客厅上墙放置旧木钟。上述内容不得加入交互实体、提示或任务条件，也不得替代床边照片、红线日记、眼镜盒和蓝色钥匙碗；茶几地毯、门垫和旧木钟均不加入碰撞。

### 4.7 首章验收

- 新玩家能在 60 秒内理解移动与交互；
- 不打开地图也能通过房间构图找到玄关；
- 提前执行操作会跳过对应教学；
- 持有条件不完整时，门给出具体、非责备提示；
- 刷新后从最近检查点恢复。

## 5. 第二章：雨中的初遇

### 5.1 地图参数

- 地图 ID：map.rain_station
- 尺寸：44×24 格
- 退化阶段：D1
- 出生点：spawn.rain.platform = (4, 19)，朝右
- 目标出口：entity.rain.clock_shop_door = (40, 5)

### 5.2 平面布局

~~~text
                         [钟表铺/红伞]
                              ▲
                    ⑤石板 ── 小巷
                      ▲         │
              雨棚 ─ ④石板 ─ 水洼
                ▲
        售票亭 ─ ②石板
          ▲
      [旧站台 P]
~~~

玩家可绕行，但正确站牌顺序会依次打开雨棚、小巷和钟表铺前的视觉通路。

背景视觉采用 30°～35° 轻俯视旧车站空间，并参考 `memory_rain_umbrella_v01.png` 的钟表铺、旧站牌、湿石材和红伞记忆氛围：左下旧站台、售票亭、中央雨棚和右上钟表铺以可见檐口、墙面厚度、站台边缘和湿地反光建立空间纵深。底图保持雨天可读，不使用黑块或重暗角；雨线与地面积水反光拆为独立环境叠层。可交互车票、石板与红伞线索仍由独立 `visual_props` 提供。

雨站运行时固定为 1280×720 单屏场景，`navigation` 范围为 x=32–1248、y=32–688。`collision` 除四侧外墙外，还覆盖左侧站房、候车亭、钟表铺与后景不可进入区域；站台临水斜边以五段阶梯式脚部碰撞近似，只阻挡角色脚底，不把视觉上的潮湿反光误判为整块实体。所有石板、车票、伞形路标和红伞均保留至少一条可到达交互环。

### 5.4 D1 触发

玩家第一次打开地图时显示完整道路。关闭地图并向右移动超过两格后进入 trigger.rain.map_wash：

1. 冻结移动 1.2 秒；
2. 播放雷声与地图纸张受潮音；
3. HUD 地图边缘出现水渍；
4. 切换 D1；
5. 显示短句“有些路名看不清了。钟声还在。”

减少动态效果模式只做 200 毫秒淡变，不使用扩散动画。

### 5.5 谜题一：站牌序列

- 谜题 ID：puzzle.rain.station_sequence
- 答案：2 → 4 → 5
- 输入：角色走入石板区域即提交
- 重置：不清空正确前缀；踩错石板变暗 600 毫秒

状态：

~~~text
[] → [2] → [2,4] → [2,4,5] → completed
~~~

若当前前缀为 [2] 而玩家踩 5，状态保持 [2]。完成后钟表铺方向的雨幕减弱，并写入 checkpoint.rain.sequence。

### 5.6 谜题二：声音路线

- 谜题 ID：puzzle.rain.sound_route
- 起点：(25, 10)
- 正确地标：红伞招牌 A → 红伞招牌 B → 钟表铺红伞
- 声音：钟声每 6 秒三响，声像指向东北
- 视觉替代：屏幕东北边缘同心圆、积水波纹朝东北扩散

玩家触碰招牌 A 后，招牌 B 才出现暖色反光；触碰 B 后开放钟表铺门前导航区。

### 5.7 提示升级

| 谜题 | 等级 1 | 等级 2 | 等级 3 |
| --- | --- | --- | --- |
| 站牌 | 车票边缘发亮 | 下一个数字与圆点发亮 | 正确石板出现水面倒影 |
| 声音路线 | 钟声视觉波纹增强 | 当前红伞招牌轻摆 | 地面积水连续显示方向 |

### 5.8 记忆与出口

与 entity.rain.red_umbrella 交互：

1. 锁定相机；
2. 播放 8 秒初遇剪影；
3. 解锁 memory.rain.umbrella；
4. 存档 checkpoint.rain.complete；
5. 红伞覆盖画面，切换第三章。

### 5.9 第二章验收

- 静音和单声道均可完成；
- 地图褪色后仍保留玩家、已到站牌和红伞标记；
- 站牌踩错不清空正确前缀；
- 从任意可走区域 45 秒内能抵达一个有效线索；
- 过场期间暂停不会破坏 D1 状态。

### 6.1 地图参数

- 尺寸：28×18 格
- 退化阶段：D2
- 出生点：spawn.life.center = (14, 14)，朝上

### 6.2 布局

~~~text
┌────────────────────────────────┐
│ 1979 同源窗  1992 同源窗  走廊  2001 同源窗 │
│ 纸箱/镜台     桂花窗台          收音机/纪念日晚餐 │
~~~

上述十个照片、相册、物件与槽位的 `visual_props` 已使用 v02 `prop_life_shared_life_atlas` Tiled tileset：frame 0–2 为不含生成式年份文字的三张照片，frame 3 为空相册，frame 4–6 为木梳、搪瓷杯、录音带，frame 7–9 为三处放置槽。所有对象保留 Tiled tile object 左下角锚点与 44×44 运行时尺寸；物件拾取后隐藏，归位后槽位实体切换到 frame 4–6。`entity.life.exit` 的正式视觉由 v02 背景中的延长走廊承担，因此不再保留伪造的 `visual_props` 占位对象。

`collision` 层使用 1280×720 运行时边界，并为相册桌、镜台、收音机柜、纪念日晚餐桌、纸箱和左下前景柜建立脚印矩形；右侧餐桌脚印为 x=960–1230、y=420–640，与收小后的桌面一致，并在收音机柜下方留出可接近槽位的通道。纸箱与镜台脚印之间保留不少于一个角色脚部宽度的通道，顶部外墙在 x=620–780 留出走廊口。`navigation` 恢复区为 x=32–1248、y=32–688，玩家不会走出可见画布。

表现层只读取既有领域状态：初始保留三段低透明度叠影；`puzzle.life.photo_order.completed` 后全部年代接缝先收束一次；`placedObjects` 中每增加一件物品，对应年代薄雾继续降低；三件全部归位后完整淡入 `environment.life.resolved`，短暂形成连贯的当代客厅印象，再由原有出口流程进入第四章。收束态保持所有碰撞对齐家具与走廊位置，只保留中间的成熟桂花窗，并移除左右时间窗、搬家纸箱、停电宴席和年代接缝。以上变化不新增进度字段，也不参与碰撞或谜题判断。

## 7. 第四章：回家的路

### 7.1 地图参数

- 地图 ID：map.return_corridor
- 尺寸：36×20 格
- 退化阶段：D3
- 出生点：spawn.return.training = (4, 10)，朝右
- 结构：训练区 + 三个独立路口房间 + 出口

### 7.2 空间结构

~~~text
[训练区] → [路口一：看见] → [路口二：记住] → [路口三：相信] → [家门]
                ↺错路房间          ↺错路房间          ↺错路房间
~~~

v02“缝隙长廊”以十字空间为稳定底板：中央与上下通道复用许家木地板和旧墙，左侧复用雨站湿石板，右侧复用共同生活章节的磨损花砖与墙面。底图不包含箭头、红伞方向、波纹或脚印答案；这些内容由运行时状态叠加。每个错路房间复用当前章节已经见过的物件组合，但停留不超过三秒，随后返回当前路口。

本轮换景不改变玩法结构。训练区、三路口顺序、世界方向答案、输入锁定、所有检查点和 D3 固定映射保持不变。

缝隙长廊运行时固定为 1280×720，`navigation` 范围为 x=32–1248、y=32–688。四个角落的实体墙分别占据上左、上右、下左、下右区域，中央十字走廊保持连通；角色可以抵达四个方向出口，但不能穿墙抄近路或走出正式背景。

### 7.3 安全教学

- 触发器：trigger.return.mapping_demo
- 输入锁定：是
- 演示时长：最短 3 秒，等待玩家确认后结束

标准模式：

1. 指南针与地砖箭头顺时针旋转；
2. 提示玩家按 W 或上键；
3. 角色安全向右移动一格；
4. 显示“方向变了。地上的箭头还在。”；
5. 玩家确认，写入 flag.return.mapping_learned。

低扰动模式：

1. 只让旧按键图标发生错位；
2. 角色仍按标准方向移动；
3. 显示“提示变得不可靠。地上的箭头还在。”。

完成后存档 checkpoint.return.training。

### 7.4 路线答案

路线以世界方向定义，输入系统负责变换。

| 路口 | 世界方向答案 | 标准模式物理输入 | 主要线索 | 通过检查点 |
| --- | --- | --- | --- | --- |
| 一 | 右 → 上 | 上 → 左 | 两枚完整地面箭头 | checkpoint.return.junction_1 |
| 二 | 下 → 右 | 右 → 上 | 第一步地面箭头、第二步伞影 | checkpoint.return.junction_2 |
| 三 | 上 → 左 → 上 | 左 → 下 → 左 | 三步哼唱波纹 | checkpoint.return.junction_3 |

低扰动模式直接使用“世界方向答案”作为物理输入。

### 7.5 路口判定

- 每走入正确出口，RouteSystem 推进一步并关闭已完成入口；
- 走入错误出口时保持当前正确前缀；
- 完整路径正确才进入下一路口；
- 玩家在同一路口停留 45 秒未移动时，播放一次环境提示；
- 暂停、打开日记和浏览器失焦不计时。

#### 状态派生线索

`returnJunction`、当前正确 `returnPrefix`、`routeLoops` 与 `hintLevel` 是第四章表现的唯一输入：

| 状态 | 当前线索 |
| --- | --- |
| 路口一，第 0/1 步 | 指向下一正确世界方向的地面箭头 |
| 路口二，第 0 步 | 指向“下”的地面箭头 |
| 路口二，第 1 步 | 指向“右”的红伞柄影子 |
| 路口三，第 0/1/2 步 | 分别指向“上 / 左 / 上”的哼唱波纹 |
| `routeLoops >= 2` 或 `hintLevel >= 2` | 沿当前正确世界方向出现淡脚印 |
| 三个路口全部完成 | 上方家门轮廓 |

线索强度按错路次数与提示等级中较高者逐级提升，但不改变方向。红伞和伞影永远只指向当前正确世界方向；错误出口不得出现红伞轮廓、红色反光或可被理解为安全锚点的伞形装饰。

### 7.6 错路与提示

| 错路次数 | 表现 |
| --- | --- |
| 1 | 当前正确线索的强度提升 |
| 2 | 正确世界方向出现淡脚印 |
| 3 | 显示明确文字提示，不显示物理按键答案 |

三级提示示例：“先朝伞柄指向的方向走。方向键和脚步的关系已经转过一圈。”

### 7.7 章节完成

通过路口三后：

1. 映射仍保持 D3；
2. 玩家只能沿直线接近家门；
3. 自动解锁 journal.return.last_page；
4. 与门交互写入 checkpoint.return.complete；
5. 淡出并切换尾声。

### 7.8 第四章验收

- 映射全章只改变一次；
- 安全教学不允许失败；
- 暂停菜单始终使用标准方向；
- 标准和低扰动模式的世界路线完全一致；
- 静音状态下第三路口仍可凭同步哼唱波纹通过；
- 错路不会清空正确前缀或返回章节开头。

### 8.3 长按区域

### 8.4 尾声验收

- HUD 消失不影响暂停和字幕；
- 长按可由无障碍设置替换为短按；
- 失焦会取消长按而不是直接完成；
- 玩家可跳过淡出动画，但不可跳过医疗免责声明；
- 科普页无需 Phaser Canvas 继续运行。
- v02 换景不改既有对白、牵手主动权、科普入口与核心四项医学结尾事实。
